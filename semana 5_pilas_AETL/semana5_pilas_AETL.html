¿Qué es una Pila? — El Principio LIFO

Una pila (stack) es un Tipo Abstracto de Dato (TAD) que organiza elementos bajo una regla muy simple: el último en entrar es el primero en salir, conocido como principio LIFO (Last In, First Out).

Piensa en una pila de platos en una cafetería. Solo puedes poner un plato encima (push) o tomar el plato de arriba (pop). Nunca puedes sacar el plato del fondo sin quitar todos los de encima. Esta restricción aparentemente simple tiene consecuencias profundas en algoritmos.

┌─────────────────────────────────────────────────────────────────────────────┐ │ PRINCIPIO LIFO — PILA DE PLATOS │ ├─────────────────────────────────────────────────────────────────────────────┤ │ │ │ ESTADO INICIAL PUSH("C") PUSH("D") POP() POP() │ │ │ │ ┌───┐ ┌───┐ │ │ ┌───┐ │ D │ ◄─top │ C │ ◄─top │ │ ┌───┐ │ C │ ◄─top ├───┤ ├───┤ │ │ │ B │ ◄─ ├───┤ │ C │ │ B │ │ │ ┌───┐◄─top ├───┤ │ B │ ├───┤ ├───┤ │ │ │ A │ │ A │ ├───┤ │ B │ │ A │ │ │ └───┘ └───┘ │ A │ ├───┤ └───┘ │ │ base base └───┘ │ A │ base devuelve "D" │ │ base └───┘ │ │ base devuelve "C" al siguiente pop │ │ │ │ REGLA: Solo se accede al elemento del TOP (cima). El resto es inaccesible.│ │ │ └─────────────────────────────────────────────────────────────────────────────┘
Explicación: La cima (top) es el único punto de acceso de la pila. Push añade encima; pop retira desde arriba. Al hacer pop sobre una pila con [A, B, C, D], se obtiene D primero, luego C, luego B, luego A — en orden inverso al de inserción. Esto es exactamente lo que necesita el historial de "deshacer": el último cambio es el primero en revertirse.

Las 5 Operaciones Fundamentales

┌─────────────────────────────────────────────────────────────────────────────┐ │ OPERACIONES DE LA PILA │ ├──────────────────┬──────────────────────────────────────────────────────────┤ │ OPERACIÓN │ DESCRIPCIÓN Y COMPORTAMIENTO │ ├──────────────────┼──────────────────────────────────────────────────────────┤ │ push(x) │ Inserta x en la cima. La pila crece. │ │ │ Pila: [A,B,C] → push("D") → [A,B,C,D] top=D │ ├──────────────────┼──────────────────────────────────────────────────────────┤ │ pop() │ Retira y devuelve el elemento de la cima. │ │ │ Pila: [A,B,C,D] → pop() → [A,B,C] devuelve D │ │ │ ⚠ ERROR si la pila está vacía (Stack Underflow) │ ├──────────────────┼──────────────────────────────────────────────────────────┤ │ peek() / top() │ Devuelve el elemento de la cima SIN eliminarlo. │ │ │ Pila: [A,B,C,D] → peek() → [A,B,C,D] devuelve D │ │ │ ⚠ ERROR si la pila está vacía │ ├──────────────────┼──────────────────────────────────────────────────────────┤ │ estaVacía() │ Devuelve true si la pila no tiene elementos. │ │ │ Siempre segura de llamar. Usar ANTES de pop/peek. │ ├──────────────────┼──────────────────────────────────────────────────────────┤ │ tamaño() │ Devuelve el número de elementos en la pila. │ │ │ Opcional: a veces se omite si no es necesaria. │ └──────────────────┴──────────────────────────────────────────────────────────┘
Explicación: Las operaciones críticas son push y pop. El error más común en producción es hacer pop() sin verificar primero estaVacía() — esto causa excepciones en tiempo de ejecución. peek() es útil cuando necesitas ver el tope sin consumirlo.

Pseudocódigo: Implementación con Lista Enlazada

Una pila puede implementarse con un arreglo o con una lista enlazada. Esta semana usaremos lista enlazada como estructura interna, porque ya la dominas y porque no requiere redimensionamiento.

La intuición clave: el tope de la pila es la cabeza de la lista enlazada. Push = insertarInicio. Pop = eliminarInicio. Ambas operaciones son O(1).

┌─────────────────────────────────────────────────────────────────────────────┐ │ PILA IMPLEMENTADA CON LISTA ENLAZADA │ │ Tope = Cabeza de la lista │ ├─────────────────────────────────────────────────────────────────────────────┤ │ │ │ clase Pila: │ │ ┌──────────────────────────────────────────────────────────────────────┐ │ │ │ tope ──────► [C|→]──►[B|→]──►[A|null] │ │ │ │ │ │ │ │ PUSH("D"): tope ──► [D|→]──►[C|→]──►[B|→]──►[A|null] │ │ │ │ (nuevo nodo apunta al tope anterior, tope = nuevo) │ │ │ │ │ │ │ │ POP(): tope ──► [C|→]──►[B|→]──►[A|null] │ │ │ │ (tope = tope.siguiente, devuelve dato del nodo viejo) │ │ │ └──────────────────────────────────────────────────────────────────────┘ │ │ │ │ Ventaja: sin límite de tamaño, push/pop siempre O(1) │ │ Desventaja: memoria extra por puntero en cada nodo │ │ │ └─────────────────────────────────────────────────────────────────────────────┘
Explicación: Al usar la cabeza de la lista como tope, push y pop son idénticos a insertarAlInicio y eliminarDelInicio. Estas operaciones son O(1) porque no requieren recorrer la lista — simplemente actualizan el puntero tope.

Pseudocódigo Comentado Línea por Línea


// ═══════════════════════════════════════════════════
//  CLASE NODO (igual que semanas anteriores)
// ═══════════════════════════════════════════════════
clase Nodo:
    dato: Tipo          // El valor que guarda este nodo
    siguiente: Nodo     // Referencia al nodo debajo en la pila (null si es el último)
    
    constructor(valor):
        this.dato = valor       // Guarda el dato recibido
        this.siguiente = null   // Por defecto, ningún nodo debajo

// ═══════════════════════════════════════════════════
//  CLASE PILA
// ═══════════════════════════════════════════════════
clase Pila:
    tope: Nodo          // Referencia al nodo en la cima (null si está vacía)
    contador: entero    // Cuántos elementos hay (opcional pero útil)
    
    constructor():
        this.tope = null        // Pila vacía: no hay tope todavía
        this.contador = 0       // Zero elementos al inicio

    // ─── PUSH: Agregar al tope ────────────────────
    metodo push(valor):
        nuevoNodo = new Nodo(valor)     // 1. Crear el nodo con el valor
        nuevoNodo.siguiente = tope      // 2. El nuevo nodo apunta al tope actual
                                        //    (si tope es null, OK: el nuevo nodo
                                        //     queda como el único)
        tope = nuevoNodo                // 3. El tope ahora es el nuevo nodo
        contador = contador + 1        // 4. Incrementar el tamaño
        // Complejidad: O(1) — no importa cuántos elementos haya

    // ─── POP: Retirar el tope ─────────────────────
    metodo pop():
        si estaVacía():                 // 1. SIEMPRE verificar antes de operar
            lanzar Error("Stack Underflow: pila vacía")
        datoCima = tope.dato            // 2. Guardar el dato a devolver
        tope = tope.siguiente           // 3. El tope ahora es el siguiente nodo
                                        //    (si era el único, tope queda null)
        contador = contador - 1        // 4. Decrementar el tamaño
        devolver datoCima               // 5. Retornar el dato guardado
        // Complejidad: O(1) — sin recorrido

    // ─── PEEK: Ver el tope sin eliminar ──────────
    metodo peek():
        si estaVacía():                 // 1. Verificar primero
            lanzar Error("Pila vacía")
        devolver tope.dato              // 2. Solo devolver el dato, no mover tope
        // Complejidad: O(1)

    // ─── ESTAVÁCIA: Consultar si hay elementos ───
    metodo estaVacía():
        devolver (tope == null)         // true si no hay nodo en el tope
        // Complejidad: O(1)

    // ─── TAMAÑO: Cuántos elementos hay ───────────
    metodo tamaño():
        devolver contador               // Solo devolver el contador ya calculado
        // Complejidad: O(1) gracias al contador
💡 ¿Por qué push apunta PRIMERO y LUEGO actualiza tope?

El orden importa: nuevoNodo.siguiente = tope debe ir ANTES de tope = nuevoNodo. Si lo haces al revés, tope ya apunta al nuevo nodo cuando intentas asignarlo, y pierdes la cadena anterior. Este error de orden es uno de los más comunes en implementaciones de pilas.

Implementaciones alternativas: Arreglo vs Lista Enlazada

Aspecto Con Arreglo estático Con Arreglo dinámico Con Lista Enlazada ✅
push() O(1) amortizado O(1) amortizado O(1) siempre
pop() O(1) O(1) O(1)
peek() O(1) O(1) O(1)
Límite de tamaño Sí (fijo) No (redimensionable) No
Memoria por elemento Solo el dato Solo el dato Dato + puntero
Localidad de caché Alta (contiguo) Alta Baja (nodos dispersos)
Implementación didáctica Simple Moderada Refuerza listas enlazadas ✅

Aplicaciones Clásicas de la Pila

🌍 ¿Dónde aparecen las pilas en software real?
  • Historial de deshacer (Ctrl+Z): Cada acción se guarda en una pila. Deshacer = pop().
  • Pila de llamadas (call stack): Cuando una función llama a otra, la dirección de retorno se guarda en la pila. Al regresar, se hace pop().
  • Verificación de paréntesis/brackets: Paréntesis abiertos se apilan; al encontrar uno cerrado, se compara con el tope.
  • Navegación web (Atrás/Adelante): Cada página visitada se guarda en una pila de historial.
  • Evaluación de expresiones: Convertir expresiones infija (2+3) a postfija (2 3 +) y evaluarlas.
  • Algoritmos de backtracking: DFS en grafos, resolución de laberintos, sudoku.

Errores Comunes Esta Semana

⚠️ Errores frecuentes que la IA podría no detectar automáticamente
  1. Orden incorrecto en push(): Asignar tope = nuevoNodo ANTES de nuevoNodo.siguiente = tope. Resultado: el nuevo nodo se apunta a sí mismo o se pierde la cadena. Diagnóstico: la pila siempre tiene 1 elemento aunque hagas varios push.
  2. No verificar estaVacía() antes de pop(): Hacer pop() sobre una pila vacía accede a null.dato, causando NullPointerException. Diagnóstico: el error ocurre solo cuando la pila ya está vacía, no siempre.
  3. Confundir peek() y pop(): Usar pop() cuando solo querías ver el tope, eliminando accidentalmente el elemento. En el verificador de paréntesis esto corrompe el resultado. Diagnóstico: la pila tiene menos elementos de los esperados.
  4. No actualizar el contador: Olvidar contador-- en pop() o contador++ en push(). La pila funciona correctamente, pero tamaño() devuelve valores incorrectos. Diagnóstico: el comportamiento es correcto pero las estadísticas de tamaño son incorrectas.

🔒 Invariantes que no deben romperse — COMPRENDE

  • Una pila vacía siempre tiene tope = null y contador = 0.
  • Si contador > 0, entonces tope ≠ null. Y viceversa.
  • El elemento devuelto por peek() SIEMPRE debe ser el mismo que devolvería pop() antes de vaciarse.
  • Después de n push seguidos de n pop, la pila debe quedar exactamente en su estado inicial.

🔒 Invariantes que no deben romperse — APLICA

  • Nunca llamar a pop() o peek() sin verificar estaVacía() primero.

La Pila como Modelo Mental

La pila no es solo una estructura de datos — es un modelo mental que aparece en múltiples capas del software. La pila de llamadas del sistema operativo, la pila de contextos en un intérprete de expresiones, la pila de estados de una interfaz con pantallas de navegación. Cuando dominas la pila, empiezas a ver este patrón en todos lados.

🤔 Reflexión crítica: ¿Cuándo la restricción LIFO es un problema?

La pila es perfecta cuando necesitas exactamente el inverso del orden de inserción. Pero si necesitas acceder al elemento más "importante" sin importar cuándo llegó, la pila falla. Para ese caso existe la Cola de Prioridad — que veremos la semana siguiente. ¿Puedes pensar en un ejemplo donde esto importa?

🧩 Extensiones Opcionales

Extensión A: Evaluación de expresiones con postfija

Las calculadoras y compiladores usan pilas para evaluar expresiones matemáticas. La expresión 3 + 4 * 2 se convierte a notación postfija 3 4 2 * + y se evalúa con una pila: al encontrar un operando, push; al encontrar un operador, pop dos operandos, opera, push resultado.

💡 Para explorar: Algoritmo Shunting Yard de Dijkstra

El algoritmo de Edsger Dijkstra convierte expresiones infijas a postfijas usando una pila. Es uno de los usos más elegantes de pilas en ciencias de la computación. Busca "Shunting Yard Algorithm" si te interesa.

Extensión B: Pila mínima (Min-Stack)

Diseña una pila que, además de las operaciones estándar, tenga un método getMin() que devuelva el mínimo elemento de la pila en O(1). Pista: necesitarás una segunda pila auxiliar.

Extensión C: DFS con pila explícita

El algoritmo de búsqueda en profundidad (DFS) en grafos o árboles funciona internamente con una pila. La versión recursiva usa la pila de llamadas del sistema; la versión iterativa usa una pila explícita. Implementa DFS en un árbol binario simple usando tu pila.

🔭 Preview de la Semana 6: Colas y Colas de Prioridad

📅 Lo que viene la próxima semana

La semana siguiente estudiarás las Colas (Queues): si la pila es LIFO, la cola es FIFO (First In, First Out). El primero en llegar es el primero en ser atendido — como una fila del banco o la cola de impresión. También explorarás la Cola de Prioridad, donde el orden no depende de cuándo llegó el elemento, sino de qué tan "importante" es. Dominar la pila esta semana es base directa para entender por qué las colas resuelven problemas que las pilas no pueden — y viceversa.

Pregunta para prepararte: ¿En qué escenarios real te importa que el primero en llegar sea el primero en ser atendido? ¿Cómo difiere eso del deshacer/rehacer que implementaste hoy?

📖 Glosario — Semana 5

Términos disciplinares y del modelo AETL

Pila (Stack)
TAD que organiza elementos bajo el principio LIFO. Solo permite acceder al elemento del tope.
LIFO
Last In, First Out. El último elemento insertado es el primero en ser retirado.
Push
Operación que inserta un elemento en el tope de la pila. Complejidad O(1) con lista enlazada.
Pop
Operación que retira y devuelve el elemento del tope. Genera error si la pila está vacía (Stack Underflow).
Peek / Top
Operación que devuelve el elemento del tope sin eliminarlo. No modifica la pila.
Stack Underflow
Error que ocurre al intentar pop() o peek() sobre una pila vacía. Debe manejarse explícitamente.
TAD (Tipo Abstracto de Dato)
Definición de una estructura por sus operaciones y comportamiento, independiente de su implementación interna.
Pila de Llamadas (Call Stack)
Estructura interna del sistema operativo que gestiona el contexto de ejecución de funciones usando el principio LIFO.
Notación Postfija
Representación de expresiones matemáticas donde el operador sigue a sus operandos: "3 4 +" = 3+4. Se evalúa eficientemente con una pila.
Agencia Compartida (AETL)
Principio donde el estudiante mantiene el rol de Arquitecto de sus productos, usando la IA como co-constructor pero sin ceder la toma de decisiones.
Caso Frontera
Situación extrema que puede romper una implementación: pila vacía, un solo elemento, o condición límite del problema.
Trade-off
Compromiso entre dos propiedades deseables que no pueden maximizarse simultáneamente (ej: velocidad vs memoria, simplicidad vs generalidad).
Verificación de Integridad
Comprobación sistemática de que los invariantes de una estructura se mantienen antes y después de cada operación.
Prompt Progresivo
Técnica AETL de formular preguntas a la IA de menor a mayor complejidad, construyendo comprensión por capas en lugar de pedir la respuesta completa de una vez.
