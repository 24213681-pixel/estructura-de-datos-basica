🔗 Conexión con la Semana 5: Pilas — Implementación y Aplicaciones

La semana pasada construiste una Pila (LIFO): el último elemento en entrar es el primero en salir. Esta semana estudiarás la Cola (FIFO), el complemento directo: el primero en entrar es el primero en salir, como una fila real. La diferencia no es trivial — mientras la pila accede a un solo extremo, la cola necesita dos: uno para entrar (enqueue) y otro para salir (dequeue). Esto introduce el desafío de la cola circular, que resolverás esta semana. Además, llevarás la idea de "orden de atención" al siguiente nivel con la Cola de Prioridad, donde el orden depende de la importancia del elemento, no de cuándo llegó. Tu Pila de la Semana 5 y tu Cola de esta semana juntas forman el HITO 1 del proyecto integrador.

🎬 ESCENARIO: El Sistema de Triage del Hospital General

Contexto: El Hospital General Regional necesita digitalizar su sistema de urgencias. La Dra. Carmen Solís, jefa de informática hospitalaria, busca un equipo de desarrollo para construir el sistema de gestión de pacientes de urgencias.
"Tenemos dos problemas críticos. Primero: la cola de recepción. Cuando un paciente llega a urgencias, entra a una fila FIFO — el primero que llegó es el primero en ser atendido, como debe ser. Pero el sistema actual en papel se corrompe fácilmente y perdemos el orden. Necesito una cola digital que no pierda datos aunque el buffer se llene."
La Dra. Solís continúa:
"Segundo, y más urgente: el triage. No todos los pacientes pueden esperar igual. Un paciente con paro cardíaco tiene prioridad sobre uno con fiebre leve, aunque el de la fiebre llegó primero. Necesito una cola donde los pacientes se ordenen por nivel de urgencia: P1 (crítico), P2 (urgente), P3 (moderado), P4 (leve). ¿Pueden construir eso?"
El analista de sistemas, Ing. Ramírez, agrega:
"El sistema también necesita el historial de atenciones del turno. Cada vez que un paciente es atendido, se registra en un log. Si hay un error médico, el auditor necesita revertir las últimas N atenciones — lo que suena exactamente a lo que hicieron con la pila la semana pasada. Así que el sistema final combina Cola FIFO para espera, Cola de Prioridad para triage, y Pila para el historial reversible."

🎯 Esta semana implementarás las estructuras de Cola y Cola de Prioridad. La integración completa con la Pila de la Semana 5 es el HITO 1.

1.1 El Principio FIFO — "El primero en llegar, primero en ser atendido"

Una Cola (Queue) es una estructura de datos lineal que sigue el principio FIFO: First In, First Out. El primer elemento que entró es el primero en salir. Piensa en la fila del banco, la impresora de la oficina, o la sala de espera del médico: nadie se "cuela" — el orden de llegada define el orden de atención.

Las dos operaciones fundamentales son:

  • Enqueue (encolar): agregar un elemento al final de la cola.
  • Dequeue (desencolar): extraer el elemento del frente de la cola.
COLA (Queue) — Principio FIFO ENQUEUE entra aquí DEQUEUE sale por aquí │ │ ▼ ▼ ┌─────────────────────────────────────────────┐ │ [D] [C] [B] [A] │ │ nuevo ───► ───► ───► frente ──► SALIDA │ └─────────────────────────────────────────────┘ final frente Estado: frente → [A] → [B] → [C] → [D] ← final Enqueue(E): frente → [A] → [B] → [C] → [D] → [E] ← final Dequeue(): retorna A frente → [B] → [C] → [D] → [E] ← final
Explicación: Este diagrama muestra la cola como un tubo: los elementos entran por un extremo (final) y salen por el otro (frente). La referencia frente apunta al próximo elemento a salir; la referencia final apunta al último elemento que llegó. Ambas referencias deben mantenerse actualizadas en todo momento. El diagrama simplifica la representación de nodos enlazados para ilustrar el flujo de datos.

1.2 Implementación con Lista Enlazada — O(1) en ambas operaciones

La forma más limpia de implementar una Cola es con una lista enlazada simple que mantiene dos referencias: frente y final. Así, tanto enqueue (insertar al final) como dequeue (eliminar del frente) operan en O(1) sin recorrer la lista.

// ══════════════════════════════════════════════════════ // PSEUDOCÓDIGO: Cola con Lista Enlazada — Comentado línea a línea // ══════════════════════════════════════════════════════ CLASE Nodo: dato // el valor almacenado siguiente ← null // puntero al siguiente nodo CLASE Cola: frente ← null // apunta al próximo elemento a salir (el más antiguo) final ← null // apunta al último elemento que entró (el más nuevo) tamaño ← 0 // opcional pero útil para saber si está llena/vacía FUNCIÓN enqueue(valor): // PASO 1: Crear el nuevo nodo nuevoNodo ← Nodo(valor) // PASO 2: Caso especial — cola vacía // Si no hay elementos, frente y final apuntan al mismo nodo SI frente == null: frente ← nuevoNodo final ← nuevoNodo SINO: // PASO 3: Cola no vacía — conectar al final // El nodo que era el último ahora apunta al nuevo final.siguiente ← nuevoNodo // PASO 4: Mover la referencia final al nuevo último nodo final ← nuevoNodo tamaño ← tamaño + 1 // Complejidad: O(1) — no recorremos nada FUNCIÓN dequeue(): // PASO 1: Verificar que no esté vacía SI frente == null: ERROR "Cola vacía — no hay qué desencolar" // PASO 2: Guardar el dato del frente antes de perderlo valorSaliente ← frente.dato // PASO 3: Mover frente al siguiente nodo frente ← frente.siguiente // PASO 4: Caso especial — la cola quedó vacía // Si frente ahora es null, final también debe serlo SI frente == null: final ← null tamaño ← tamaño - 1 RETORNAR valorSaliente // Complejidad: O(1) — solo movemos un puntero FUNCIÓN verFrente(): // Solo consultar sin extraer SI frente == null: ERROR "Cola vacía" RETORNAR frente.dato // Complejidad: O(1) FUNCIÓN estaVacia(): RETORNAR frente == null

🔒 Invariantes que no deben romperse — Cola con Lista Enlazada

  • Si estaVacia() es verdadero, entonces ambos punteros, frente y final, deben ser null. Si solo uno es null, hay un bug.
  • Si la cola tiene exactamente un elemento, frente y final deben apuntar al mismo nodo.
  • final.siguiente debe ser siempre null. Si no lo es, la lista tiene un ciclo o un error de enlace.
  • El tamaño nunca puede ser negativo. Si decrece por debajo de 0, se llamó dequeue() sin verificar la cola vacía.

1.3 El Problema del Arreglo y la Cola Circular

Si implementas una cola con un arreglo fijo, surge un problema clásico: aunque hagas dequeue y "liberes" el espacio del frente, el índice frente avanza hacia la derecha y ese espacio queda inutilizable. Eventualmente la cola reporta "llena" aunque haya espacio libre al inicio. La solución es la Cola Circular.

PROBLEMA DEL ARREGLO LINEAL (sin circular): Índices: [0] [1] [2] [3] [4] [ ] [ ] [C] [D] [E] ↑ ↑ frente final ¿Podemos hacer enqueue(F)? → final está en [4] = último índice → "Cola llena" → Pero [0] y [1] están libres (dequeue los limpió) → ¡Espacio desperdiciado! Esto es el problema. SOLUCIÓN: COLA CIRCULAR (con arreglo de capacidad 5) Los índices se envuelven: después del [4] viene el [0] Fórmula: nuevoIndice = (indiceActual + 1) % capacidad Estado inicial: Índices: [0] [1] [2] [3] [4] [F] [ ] [C] [D] [E] ↑ ↑ final frente Después de enqueue(F): final avanzó de [4] a [0] usando módulo → (4 + 1) % 5 = 0 ✓ El espacio libre se reutiliza
Explicación: El diagrama muestra cómo el arreglo lineal desperdicia espacio al desencolar (los primeros índices quedan libres pero inaccesibles). La cola circular resuelve esto usando aritmética modular: cuando el índice llega al final del arreglo, "da la vuelta" al inicio. Esto requiere manejar con cuidado la condición de "cola llena" vs "cola vacía", ya que en ambos casos frente == (final + 1) % capacidad puede confundirse.

1.4 Cola de Prioridad — Cuando el orden de llegada no es suficiente

Una Cola de Prioridad no respeta el orden de llegada: el elemento con mayor prioridad (o menor, según el criterio) siempre sale primero, independientemente de cuándo llegó. Es como la sala de urgencias: el paciente con paro cardíaco pasa antes que el de la fractura, aunque llegó después.

La implementación más eficiente de una Cola de Prioridad usa un Heap Binario:

  • Min-Heap: el elemento de menor valor (mayor urgencia si los números son menores) está siempre en la raíz.
  • Max-Heap: el elemento de mayor valor está en la raíz.
MIN-HEAP BINARIO — Triage Hospitalario (prioridad 1 = crítico) [P1: García] / \ [P2: López] [P1: Martínez] / \ / [P3: Cruz] [P2: Ríos] [P4: Díaz] Propiedad del heap: el padre siempre tiene prioridad ≤ que sus hijos (en un min-heap). El nodo raíz siempre es el de mayor urgencia. Representación en arreglo (más común): Índice: [0] [1] [2] [3] [4] [5] Dato: [P1:Grc] [P2:Lpz] [P1:Mrt] [P3:Cruz] [P2:Ríos] [P4:Díaz] Para el nodo en índice i: → Hijo izquierdo: índice 2i + 1 → Hijo derecho: índice 2i + 2 → Padre: índice (i - 1) / 2
Explicación: El heap binario se visualiza como un árbol completo donde cada nodo padre tiene prioridad menor o igual a sus hijos (min-heap). Lo notable es que se puede representar eficientemente en un arreglo sin necesidad de punteros — la posición de los nodos está determinada matemáticamente por sus índices. Las operaciones clave son sift-up (después de insertar, el nuevo elemento "sube" hasta su posición correcta) y sift-down (después de extraer la raíz, el último elemento "baja" a su posición). Ambas son O(log n).

3.1 Trade-offs: Lista ordenada vs Heap para Cola de Prioridad

El Reto 2 te pidió implementar la Cola de Prioridad con inserción ordenada en lista enlazada. Esto es conceptualmente claro pero tiene un costo: cada enqueue() puede recorrer toda la lista para encontrar la posición correcta — O(n). El heap binario, en cambio, mantiene solo la propiedad del padre-hijo, logrando O(log n) para ambas operaciones. La pregunta real no es "¿cuál es mejor?" sino "¿cuál es mejor para este problema específico?"

Criterio Lista enlazada ordenada Heap binario
enqueue() O(n) — recorre para insertar en posición O(log n) — sift-up
dequeue() O(1) — siempre extrae la cabeza O(log n) — sift-down después de extraer raíz
Complejidad conceptual Baja — fácil de entender y depurar Media — requiere entender árbol + arreglo
¿Cuándo usar? Colas pequeñas (<100 elementos), prototipado Colas grandes, sistemas en producción
¿Se mantiene orden FIFO dentro del mismo nivel? ✅ Sí — insertar con <= (no <) garantiza que los elementos de igual prioridad se mantengan en orden de llegada sin esfuerzo adicional. ⚠️ No automáticamente — el heap no conserva orden de inserción entre elementos de igual prioridad. Requiere un campo adicional ordenLlegada (timestamp o contador) como criterio de desempate secundario, lo que aumenta la complejidad del comparador.

→ Si falla: revisa el caso especial "cola vacía" en el método enqueue. frente y final deben inicializarse al mismo nodo nuevo.

Casos de Prueba Detallados

✅ Caso 1 — Operaciones básicas FIFO

Secuencia: enqueue(1) → enqueue(2) → enqueue(3) → dequeue() → verFrente()
Resultado esperado: dequeue() retorna 1, verFrente() retorna 2, tamaño = 2
Qué verifica: Orden FIFO correcto y que verFrente() no extrae.

✅ Caso 2 — Cola vacía con múltiples ciclos

Secuencia: enqueue(A) → dequeue() → enqueue(B) → dequeue() → estaVacia()
Resultado esperado: estaVacia() = verdadero, tamaño = 0, frente = null, final = null

✅ Caso 3 — Triage con 5 pacientes

Secuencia de llegada: Ana(P3), José(P1), María(P2), Carlos(P4), Luisa(P1)
Orden esperado de atención: José(P1), Luisa(P1), María(P2), Ana(P3), Carlos(P4)

⚠️ Caso 4 — Casos frontera críticos

Prueba A: dequeue() en cola vacía → debe lanzar error, no colapsar.
Prueba B: enqueue seguido inmediato de dequeue (un solo elemento) → frente y final deben ser null después.
Prueba C: enqueue(X, P1) cuando ya hay un P1 en cola → verifica que X quede DESPUÉS del P1 existente (FIFO dentro del mismo nivel).

🔴 Errores Comunes Esta Semana

1. Olvidar actualizar final = null cuando la cola queda vacía: frente avanza a null pero final sigue apuntando al nodo eliminado. El siguiente enqueue crea un ciclo fantasma.

2. Confundir FIFO con LIFO en la Cola de Prioridad: La prioridad define quién pasa primero en su nivel, pero dentro del mismo nivel debe respetarse el orden de llegada. La IA frecuentemente no maneja el empate FIFO correctamente.

3. Usar la cola circular con el arreglo pero sin el módulo: El índice final llega al último elemento y el sistema reporta "lleno" aunque haya espacio libre al inicio. La aritmética modular no es opcional.

4. No verificar estaVacia() en dequeue(): Desencolar de una cola vacía resulta en acceso a null.dato — error de puntero nulo que la IA frecuentemente no coloca en el código generado a menos que se lo pidas explícitamente.

📖 Glosario — Semana 6

Cola (Queue)
Estructura de datos lineal que sigue el principio FIFO. Dos extremos: frente (salida) y final (entrada). Operaciones principales: enqueue y dequeue.
FIFO (First In, First Out)
Principio de ordenamiento donde el primer elemento que entró es el primero en salir. Contraste con LIFO de las pilas.
Enqueue
Operación de inserción en una cola: agrega un elemento al extremo final. O(1) con lista enlazada manteniendo referencia al final.
Dequeue
Operación de extracción en una cola: elimina y retorna el elemento del extremo frente. O(1) con lista enlazada.
Cola Circular
Implementación de cola con arreglo de tamaño fijo donde los índices se envuelven usando aritmética modular: (índice + 1) % capacidad. Resuelve el desperdicio de espacio del arreglo lineal.
Aritmética Modular
Operación % (módulo) que retorna el residuo de una división. Fundamental para la cola circular: permite que el índice "dé la vuelta" al inicio del arreglo.
Cola de Prioridad (Priority Queue)
Estructura donde cada elemento tiene una prioridad asociada. El elemento de mayor prioridad siempre sale primero, independientemente del orden de llegada.
Heap Binario
Árbol binario completo donde cada padre tiene prioridad ≤ (min-heap) o ≥ (max-heap) que sus hijos. Representable eficientemente en arreglo. Base de la Cola de Prioridad eficiente.
Min-Heap
Heap donde el elemento de menor valor está siempre en la raíz. Usado cuando "menor número = mayor prioridad" (ej: P1 crítico < P4 leve).
Sift-Up / Sift-Down
Operaciones de restauración del heap: sift-up sube un elemento hasta su posición correcta tras inserción; sift-down baja la nueva raíz hasta su posición correcta tras extracción. Ambas O(log n).
Deque (Double-Ended Queue)
Cola de doble extremo: permite insertar y extraer por ambos lados en O(1). Generaliza tanto la pila como la cola. Implementación natural con lista doblemente enlazada.
Triage
Proceso de clasificación de pacientes por urgencia médica. Contexto del escenario que motiva la Cola de Prioridad: P1 (crítico) siempre antes que P4 (leve).
Agencia Compartida (AETL)
Principio donde el estudiante mantiene el rol de Arquitecto. En esta semana: el estudiante decide si usar lista ordenada o heap, cómo manejar empates, y cómo integrar las tres estructuras.
Trade-off
Compromiso entre propiedades deseables. Esta semana: lista ordenada (enqueue O(n), dequeue O(1), simple) vs heap (ambas O(log n), más complejo). No hay respuesta única correcta — depende del contexto.
Caso Frontera
Situación límite que puede romper una implementación: cola vacía, un solo elemento, empates de prioridad, buffer lleno en cola circular. Siempre deben probarse explícitamente.
Invariante de Estructura
Condición que debe ser verdadera siempre en la estructura. En la Cola: si está vacía, ambos frente y final son null. Si tiene un elemento, frente y final apuntan al mismo nodo.
